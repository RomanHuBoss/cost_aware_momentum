# Patch 1.2.2

Исправляет одновременное отображение нескольких актуальных рекомендаций для одного и того же символа.

## Причина

Часовой inference выполняется раз в 60 минут, а срок действия рекомендации по умолчанию равен 90 минутам. Поэтому сигнал предыдущего часа еще оставался `PUBLISHED`, когда уже создавался следующий. API возвращал обе записи, и операторская панель показывала две плитки одного символа.

## Что изменено

1. Перед публикацией нового сигнала worker атомарно переводит предыдущий `PUBLISHED`-сигнал того же символа в `SUPERSEDED`.
2. Ожидающие действия execution plans старого сигнала также переводятся в `SUPERSEDED`.
3. Планы со статусами `ACCEPTED`, `ENTERED`, `PARTIAL`, `CLOSED`, `REJECTED` и `EXPIRED` сохраняются для журнала и учета открытой позиции.
4. В PostgreSQL добавлен частичный уникальный индекс: один `PUBLISHED`-сигнал на символ.
5. API по умолчанию выбирает только наиболее свежую рекомендацию каждого символа.
6. Frontend дополнительно устраняет дубликаты по `expires_at` во время rolling upgrade.
7. Устаревшую рекомендацию нельзя принять или пересчитать как текущую.
8. Добавлен аудит `MARKET_SIGNAL_SUPERSEDED` и ссылка на заменяющий сигнал в ответе stale accept.

## Установка

Остановите приложение, замените файлы проекта и выполните:

```powershell
py -3.12 manage.py migrate
py -3.12 manage.py doctor
py -3.12 manage.py run
```

Миграция автоматически:

- оставит самый свежий `PUBLISHED`-сигнал каждого символа;
- переведет более старые пересекающиеся сигналы и ожидающие планы в `SUPERSEDED`;
- создаст защитный уникальный индекс.

Изменять `SIGNAL_TTL_MINUTES=90` не требуется: перекрытие срока действия допустимо, но в операторской панели текущей считается только последняя рекомендация символа.

## Проверка

После запуска запрос:

```text
GET /api/v1/recommendations
```

должен возвращать не более одной карточки на каждый `symbol`. В ответе присутствует поле:

```json
"latest_per_symbol": true
```
